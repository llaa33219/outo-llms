# BLP TILE

> A window-container design language inspired by tiling window managers.
> **BLP TILE은 레이아웃/컨테이너 규약만 정의한다.** 창 *안쪽*의 비주얼은 다른 디자인 언어가 책임진다.

---

## 1. 개요

BLP TILE은 데스크탑 tiling window manager(i3, bspwm, dwm, Hyprland 등)에서 아이디어를 가져온 **웹/앱/영상 등 다양한 용도의 컨테이너 디자인 언어**다. 화면 위의 모든 요소를 **"창(Window)"** 이라는 단위로 추상화하고, 이 창들이 어떤 규칙으로 배치·상호작용하는지만 정의한다.

### 의도

- **레이아웃을 "창들의 관계"로 사고하자.** 절대 좌표, 매직 넘버, 컨테이너 컴포넌트 중첩에서 벗어나, *이 창은 동적, 저건 플로팅* 같은 선언만으로 화면이 구성되도록.
- **컨테이너는 자기가, 내용은 다른 디자인 언어가.** BLP TILE은 창 자체의 형태·배치·상태·간격만 책임지고, 그 안에 들어가는 버튼·텍스트·차트 등은 다른 디자인 언어(Material, shadcn, Ant Design, 자체 가이드 등)가 담당한다. **반드시 다른 디자인 언어와 함께 사용해야 정상 동작한다.**

### 하지 *않는* 것

- 창 끄기, 최소화, 최대화, 드래그 이동, 키보드로 타일링 전환 같은 WM의 조작 기능은 **기본 범위 밖**이다. 기본 창에는 close/minimize/fullscreen 버튼과 title bar도 없다 — 자세한 chrome 정책은 §5.1. 필요하면 의도에 따라 추가할 수 있지만, 일반적인 웹/앱 디자인에서는 거의 쓰이지 않는다.

## 2. 핵심 컨셉

```
┌─ Tiling Space ──────────────────────────────────────┐
│                                                     │
│  ┌────────┐  ┌────────────────┐  ┌──────────────┐   │
│  │ TopBar │  │  Chat History  │  │ Notifications│   │  ← 동적/정적 창들 (4px gap)
│  │        │  │                │  │              │   │
│  └────────┘  │                │  └──────────────┘   │
│  ┌────────┐  │                │  ┌──────────────┐   │
│  │ Input  │  └────────────────┘  │ Document     │   │
│  │        │  ┌────────────────┐  │              │   │
│  └────────┘  │   Chat Input   │  └──────────────┘   │
│              └────────────────┘                     │
│                                                     │
│        ┌──────────────────┐                         │  ← 완전 플로팅 (로그인 모달)
│        │   Login Modal    │                         │
│        └──────────────────┘                         │
└─────────────────────────────────────────────────────┘
```

- 모든 UI 요소는 **창(Window)** 안에 있다. 텍스트, 아이콘, 이미지도 자기 창을 가진다.
- 창은 **타입**을 가지며, 타입이 곧 배치·스크롤·z-index 정책을 결정한다.
- 창과 창 사이는 항상 **4px gap**.

## 3. 토큰 (Design Tokens)

| Token | 값 | 용도 |
|---|---|---|
| `--blp-gap` | `4px` | 창과 창 사이 간격 |
| `--blp-border-width` | `4px` | 포커스/비포커스 테두리 두께 |
| `--blp-border-focus` | `#007bff` | 호버/포커스된 창의 테두리 색 |
| `--blp-border-idle` | `#94989e` | 비활성 창의 테두리 색 |
| `--blp-radius` | `12px` | 창의 모서리 둥글기 (Border 레이어에 적용) |
| `--blp-z-base` | `0` | 동적/정적 창 기본 z-index |
| `--blp-z-fixed-float` | `50` | 고정 플로팅 |
| `--blp-z-full-float` | `100` | 완전 플로팅 (모달 등) |

> **테두리 두께 4px는 박스 모델에 포함한다.** 즉 `--blp-border-width`만큼 안쪽 콘텐츠가 줄어든다. 테두리 두께를 기준으로 캔버스를 쪼개야 하기 때문(타일링의 핵심).

## 4. 창 타입 (Window Types)

창은 **4가지 타입** 중 하나를 가진다. 타입은 한 번 정해지면 그 창의 수명 동안 변하지 않는다.

### 4.1 동적 (Dynamic)

실제 tiling WM처럼 **부모 공간 안에서 자율적으로 재배치**되는 창.

- 부모 컨테이너가 정한 tiling space 안에서 형제 창들과 협상해 자기 영역을 잡는다.
- 형제 창이 추가/제거/리사이즈되면 자동으로 reflow한다. 특히 **동적 창이 제거되면 인접 동적 창이 그 자리를 채운다** (정적·플로팅은 움직이지 않음). 리플로우 애니메이션은 §7.5.
- 일반적인 split: 좌우(bsp) 또는 상하. 필요시 더 세분화 가능.
- 가로/세로 split은 `weight`로 비율을 갖는다 (예: `2 : 1`).

```
Parent: Tiling Space (가로 100%)
┌────────────┬────────────────┐
│  Dynamic A │   Dynamic B    │   weight = 1 : 2
│  (Sidebar) │   (Main)       │
│            │                │
└────────────┴────────────────┘
        ↑ 닫으면 B가 자동 확장
```

**쓰는 곳**: 사이드바, 메인 패널, 채팅 히스토리/입력 등 화면 구도의 뼈대.

### 4.2 정적 (Static)

**한 번 배치되면 자기 위치/크기를 유지**하는 창. 동적 창과는 다르게 reflow하지 않는다.

- 단, 형제 동적 창은 정적 창의 영역을 **회피**한다. (gap, 인접 거리 등 상호작용함)
- 즉 "타일링 안에서 떠 있는 고정 앵커"다. 위치와 크기는 명시적으로 지정.

```
Tiling Space
┌────────┬───────────────────────┐
│ Dyn A  │      Dyn B            │
│        │                       │
│        ├───────────┬───────────┤
│        │  Static C │   Dyn D   │   ← C는 고정, B와 D는 C를 피해 흐른다
└────────┴───────────┴───────────┘
```

**쓰는 곳**: 툴 팔레트, 미니맵, 항상 같은 자리의 컨트롤 패널 등.

### 4.3 고정 플로팅 (Fixed Floating)

**문서/콘텐츠 흐름 위에 떠 있되, 스크롤하면 같이 움직이는** 창.

- z-index가 동적/정적 창보다 높다.
- `position: absolute` 처럼 동작 — 부모 영역 안에서의 좌표를 가지며, 스크롤 시 함께 이동.
- 콘텐츠를 *읽는 동안* 따라다녀야 하는 보조 정보에 적합.

**쓰는 곳**: 챕터 내 코드 카피 버튼, 인라인 툴팁, 푸터 위 플로팅 액션바 등.

### 4.4 완전 플로팅 (Full Floating)

**뷰포트에 고정**되어, 스크롤과 무관하게 항상 같은 화면 위치에 있는 창.

- `position: fixed` 처럼 동작.
- z-index는 고정 플로팅보다 같거나 높다.
- 모달, 토스트, 알림 센터, 글로벌 단축키 안내창 등 **"지금 이 순간 사용자에게 보여야 하는 것"**에 사용.

**쓰는 곳**: 로그인 모달, 글로벌 알림, 시스템 메시지, 확인 다이얼로그.

### 4.5 한눈에 비교

| 타입 | Reflow | 스크롤 추적 | z-index | 비유 |
|---|---|---|---|---|
| 동적 (Dynamic) | ✅ 자동 | 부모 따라감 | base | i3의 tiled 윈도우 |
| 정적 (Static) | ❌ 고정 | 부모 따라감 | base | i3의 floating in tiled |
| 고정 플로팅 (Fixed Floating) | ❌ | ✅ 따라감 (absolute) | ↑ | sticky/absolute |
| 완전 플로팅 (Full Floating) | ❌ | ❌ 뷰포트 고정 (fixed) | ↑↑ | modal/dialog |

## 5. 창 구조 (Anatomy)

모든 창은 다음 레이어로 구성된다. *안쪽 레이어의 스타일은 BLP TILE이 정하지 않는다* — 다른 디자인 언어로 채운다.

```
┌─ BLP TILE Window ─────────────────────────────┐
│  ┌─ Border (4px, focus/idle) ───────────────┐ │
│  │  ┌─ Inner Surface ─────────────────────┐ │ │
│  │  │  (다른 디자인 언어가 채우는 영역)   │ │ │
│  │  │   - 헤더, 본문, 푸터, 컴포넌트 …    │ │ │
│  │  └─────────────────────────────────────┘ │ │
│  └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

- **Border**: 4px solid, focus면 `#007bff`, 아니면 `#94989e`. *창의 현재 상태*만 표현한다. 모서리는 `12px` 둥글림(`--blp-radius`).
- **Inner Surface**: 창의 실제 콘텐츠. 여기부터는 다른 디자인 언어의 영역이다. 모서리 둥글림은 Border와 동일하게 `12px`.

창 안쪽에 들어가는 요소들 — 상단 바, 채팅 히스토리, 채팅 입력, 로그인 폼, 하단 바, 알림, 문서 뷰어, 이미지 등 — 은 **각자 자기만의 디자인 언어를 가져도 되고, 부모 창과 같은 걸 써도 되고, 통째로 외부에서 import 해도 된다.** BLP TILE은 그 선택에 간섭하지 않는다.

### 5.1 기본 크롬 없음 (Chrome-less by default)

BLP TILE의 창은 **기본적으로 OS/데스크탑 창이 가진 chrome을 갖지 않는다.** Inner Surface는 곧바로 콘텐츠가 채우는 평면이다.

**기본으로 존재하지 않는 것**:
- ❌ 닫기 버튼 (✕)
- ❌ 최소화 버튼 (—)
- ❌ 전체화면 버튼 (□)
- ❌ 창 제목이 표시되는 상단 바 (title bar)
- ❌ 아이콘, 시스템 메뉴 등 OS window chrome 전반

**Inner Surface의 기본 상태**:
- 콘텐츠가 Border 안쪽 **100%**를 차지한다. 상하좌우에 chrome 영역이 따로 없다.
- "창의 정체성"은 BLP TILE의 Border + (필요 시) 다른 디자인 언어의 *콘텐츠 헤더*로 표현한다. OS식 title bar로 표현하지 않는다.
- "콘텐츠 헤더"는 앱 내부 컴포넌트(예: 페이지 타이틀, 탭 이름)이지, OS 윈도우 chrome이 아니다. 두 개념을 섞지 않는다.

**예외 — 명시적으로 켤 수 있는 케이스**:
- **이동 가능한 창**이 필요할 때: 드래그 핸들 + 최소한의 컨트롤 노출
- **다른 스타일 시스템 복제**가 필요할 때: GTK, Qt, macOS, Windows 등 특정 OS/툴킷의 chrome을 *의도적으로* 재현
- **브라우저 멀티 윈도우**: `window.open` 등으로 생성된 진짜 OS 창을 BLP TILE 컨테이너로 다룰 때

예외는 *명세에 "이 창은 chrome 포함" 등으로 명시*되어야 기본값을 덮어쓴다. 별도 선언이 없으면 chrome-less가 정답이다.

## 6. 상태 & 인터랙션

### 6.1 Focus / Idle

- **현재 마우스가 올라가 있는 창** = Focus. 테두리 `4px solid #007bff`.
- **나머지 모든 창** = Idle. 테두리 `4px solid #94989e`.
- 키보드 포커스도 동일 규칙으로 매핑하는 것을 권장(필수는 아님).

### 6.2 Gap 규칙

- 인접한 두 창의 경계 사이 거리는 항상 **4px**.
- 동적 창끼리 reflow할 때도, 정적 창을 동적 창이 회피할 때도, 플로팅 창의 앵커링 기준도 모두 4px가 기본.
- 다중 행/열 tiling에서도 셀과 셀 사이는 4px.

### 6.3 Z-Index

| 타입 | z-index |
|---|---|
| 동적 / 정적 | base (`0` ~ 컨테이너 안에서) |
| 고정 플로팅 | `50` |
| 완전 플로팅 | `100` |

플로팅 창끼리 겹치면 선언 순서대로 쌓는다(나중에 선언된 것이 위).

### 6.4 기본 인터랙션 범위

- 클릭/호버로 focus 변경 ✅
- 드래그로 창 이동 ❌ (기본)
- 창 닫기/최소화 ❌ (기본)
- 키보드 단축키로 tiling 모드 변경 ❌ (기본)

위 항목들은 **의도적으로 명시하면** 추가할 수 있다. 예를 들어 "이 앱은 창을 끌 수 있다"가 명세에 있다면 켜고, 아니면 끄는 게 BLP TILE의 정통 사용법이다.

## 7. 애니메이션

BLP TILE의 창은 **언제, 어떻게** 나타나고 사라지는지가 정해져 있다. 콘텐츠 자체의 모션(버튼 hover, 리스트 stagger 등)은 다른 디자인 언어가 책임지므로 여기서는 다루지 않는다.

#### 책임 경계 — 각자 자기 영역 안에서

BLP TILE의 애니메이션은 **창(Window) 자체**에 적용된다. **창 안쪽**의 모션은 다른 디자인 언어가 자기 방식으로 담당한다. 두 레이어는 서로를 막지 않는다.

- **BLP TILE**: 창 레벨 Enter/Exit (§7.1) + Reflow (§7.5) — 창 전체(보더 + Inner Surface)에 적용
- **다른 디자인 언어**: 창 *안쪽* 콘텐츠의 등장/퇴장/라우트 전환, hover, 마이크로 인터랙션 등 — 자기 영역 안에서 자유롭게

**창 안에서는 다른 디자인 언어의 등장/퇴장/라우트 애니메이션을 그대로 써도 된다.** BLP TILE이 그것들을 막거나 무시하지 않는다. 두 레이어의 애니메이션은 각자 자기 책임을 진다.

- BLP TILE: "창이 들어왔다/나갔다"는 *컨테이너 차원*의 신호
- 다른 디자인 언어: "안의 콘텐츠가 바뀌었다"는 *콘텐츠 차원*의 신호

**기본값은 둘 다 켜는 것이다.** 이중 모션(창이 들어오는 동안 내부 라우트도 fade-in 등)이 의도적으로 싫으면 명세에 선언해 한쪽을 끌 수 있다. 예: "이 앱은 내부 라우트 전환 모션을 끄고 BLP TILE 창 전환만 사용한다" 또는 그 반대. 별도 선언이 없으면 둘 다 그대로 동작한다.

### 7.1 Enter / Exit

창의 등장(Enter)과 퇴장(Exit)은 항상 다음 규칙을 따른다.

| 동작 | 변화 | 이징 | 지속시간 |
|---|---|---|---|
| **Enter** | scale 0.8 → 1.0, opacity 0 → 1 | `easeOutBack` | 100ms |
| **Exit**  | scale 1.0 → 0.8, opacity 1 → 0 | `easeInBack`  | 100ms |

핵심:
- **Exit = easeInBack** — 끝부분에서 살짝 "들렸다가" 빠지는 느낌. 명시적인 종료 신호.
- **Enter = easeOutBack** — 살짝 "튀어오르듯" 자리 잡는 느낌. 자연스러운 출현.
- 두 동작 모두 **0.1s**. 빠르게 처리해서 인터랙션 흐름을 끊지 않는다.
- scale과 opacity는 **동시에** 진행한다. 한쪽만 지연시키지 않는다.

```
Exit (100ms, easeInBack)            Enter (100ms, easeOutBack)

  scale 1.0 ▓▓▓▓▓░░░░  0.8          scale 0.8 ░░░░▓▓▓▓▓  1.0
  opa   1.0 ▓▓▓▓▓░░░░  0.0          opa   0.0 ░░░░▓▓▓▓▓  1.0
        t=0 ──────── t=0.1              t=0 ──────── t=0.1
```

### 7.2 첫 등장 — Stagger

"맨 처음" 화면이 구성될 때 — 즉 사용자가 처음 진입한 시점 — 모든 창은 **동시에** 등장하지 않는다. 같은 easing/duration이라도 **시작 시점을 살짝씩 어긋난 stagger**로 등장한다.

- 권장 stagger 간격: **30~50ms / 창**
- 등장은 부모(가장 바깥 tiling space)부터 자식으로, 위→아래·좌→우 순으로.
- Stagger는 **Enter 애니메이션의 시작 시점만** 미룬다. duration/easing/scale 범위는 7.1과 동일.

```
[Win A: t=0.00] [Win B: t=0.04] [Win C: t=0.08] [Win D: t=0.12]
     ▼              ▼              ▼              ▼
   100ms          100ms          100ms          100ms
 (easeOutBack)  (easeOutBack)  (easeOutBack)  (easeOutBack)
```

**적용 조건** — 다음 중 하나일 때 "첫 등장"으로 본다.
- 페이지/앱 첫 로드
- Tiling Space 자체가 새로 마운트되는 시점

**적용하지 않는 조건** — 다음은 "첫 등장"이 아니다.
- 같은 화면에서의 탭 전환
- 같은 화면에서의 페이지 라우팅
- 모달/드로어 등 부분적 오버레이의 열기 (오버레이 자체만 7.1 규칙 적용)

#### 동적 창 등장 — Reflow 결합

첫 등장 시점에 **동적 창**이 새로 등장하면, 이미 떠 있던 **동적 창들**이 §7.5의 reflow 규칙으로 자동으로 자리를 비켜준다. 즉 "첫 등장 stagger"와 "reflow"가 결합된 결과다.

**순서**:
1. 첫 동적 창 등장 → tiling space를 100% 독점 (가장 큰 크기)
2. 둘째 동적 창 등장 (stagger delay 후) → 첫째가 reflow로 줄어들고, 둘째가 그 자리에 등장
3. 셋째 동적 창 등장 → 첫째+둘째 모두 reflow로 줄어들고, 셋째가 등장
4. ... N번째까지 반복, weight 비율에 따라 최종 분배

```
t=0.00: ┌────────── A ──────────┐
        A enters, 100% 공간 독점

t=0.04: ┌──── A ────┬─── B ───┐
        A reflow     B enters
        (3fr → 1.5)  (1.5fr)

t=0.08: ┌─ A ─┬── B ──┬─ C ─┐
        A,B reflow   C enters
        (1.5 → 1fr)  (1fr)

t=0.12: ┌A─┬B─┬C─┬D─┐
        A,B,C reflow, D enters
        (1 → 0.75fr each)

완성: 0.75fr × 4 = 3fr
```

핵심:
- 새 동적 창의 **Enter** (100ms easeOutBack)와 기존 동적 창의 **Reflow** (각 태두리별 100ms easeOutBack, §7.5)가 **동시에** 진행된다. "들어오는 동안 옆으로 비켜난다"는 시각 효과.
- 정적·플로팅 창은 reflow 대상이 아니므로 등장도 안 하고 움직이지도 않음. 이들이 tiling space에 함께 있으면 동적 창의 가용 영역이 줄어든 채로 등장한다.
- §7.5의 "각 태두리별 0.1s easeOutBack" 규칙이 그대로 적용된다.

### 7.3 재등장 — Diff 기반

탭 전환, 페이지 이동처럼 **창 안의 콘텐츠가 다른 요소로 교체**될 때, 모든 창을 재등장시키지 않는다.

원칙:
- **요소 내부 변화** (텍스트 변경, 카운트 증가, 아이콘 교체 등) → 애니메이션 없음. 다른 디자인 언어의 영역.
- **요소 자체가 바뀜** (탭 A → 탭 B, 페이지 1 → 페이지 2) → *바뀐 요소만* Enter/Exit 규칙을 적용해 등장/퇴장.
  - 사라지는 요소: **Exit** (100ms, easeInBack)
  - 새로 들어오는 요소: **Enter** (100ms, easeOutBack)
  - **남아있는 요소는 가만히 있는다.** 재등장 ❌, 리플로우만 ⭕
- 같은 위치에 다른 요소가 들어와도 **stagger는 적용하지 않는다.** (Stagger는 7.2의 "첫 등장" 전용.)

```
Tab A → Tab B
─────────────────────────────────────────────────────
[A1: stay] [A2: stay] [A3: stay] [B1: enter] [A4: stay] [B2: enter]
   같은 요소, 가만히           사라짐(A2)         새 요소만 등장
                                       [B3: enter]
```

**판단 기준 요약**:
- "이 자리에 아까 그게 있었나?" → **No** → Enter
- "이 자리에 있던 게 지금 사라졌나?" → **Yes** → Exit
- 둘 다 No → 정적, 애니메이션 없음

### 7.4 토큰

| Token | 값 | 용도 |
|---|---|---|
| `--blp-anim-duration-enter` | `100ms` | Enter 지속시간 |
| `--blp-anim-duration-exit`  | `100ms` | Exit 지속시간 |
| `--blp-anim-ease-enter`     | `easeOutBack` | Enter 이징 |
| `--blp-anim-ease-exit`      | `easeInBack`  | Exit 이징 |
| `--blp-anim-scale-from`     | `0.8`  | Enter 시작 / Exit 종료 시 scale |
| `--blp-anim-scale-to`       | `1.0`  | Enter 종료 / Exit 시작 시 scale |
| `--blp-anim-stagger`        | `40ms` | 첫 등장 stagger 간격 (권장 30~50ms) |
| `--blp-anim-duration-reflow` | `100ms` | 리플로우 변 애니메이션 지속시간 (각 태두리) |
| `--blp-anim-ease-reflow`     | `easeOutBack` | 리플로우 변 애니메이션 이징 |

### 7.5 Reflow — 창 제거 & 크기 전환

동적 창은 형제 동적 창의 추가/제거/리사이즈에 반응해 **자율적으로 크기를 재조정**한다. 정적·플로팅 창은 이 리플로우에 참여하지 않는다 — 자기 자리를 유지한다.

**핵심 규칙 — 제거 시 채우기**

- 동적 창이 **제거**되면, 인접 동적 창이 그 자리를 자동으로 채운다.
- 예: 정적 상단바 + 동적 메뉴 + 동적 뷰어 구성에서 뷰어가 제거되면, 메뉴가 늘어나서 뷰어의 자리를 가져간다.

```
[Before]                                 [After — Viewer 제거]
┌─ TopBar (Static) ─────────────────┐    ┌─ TopBar (Static) ─────────────────┐
├──────────┬──────────────────────────┤    ├─────────────────────────────────────┤
│   Menu   │        Viewer            │    │                Menu                 │
│ (Dynamic)│       (Dynamic)          │ →  │              (Dynamic)              │
│   1fr    │         2fr              │    │                3fr                  │
└──────────┴──────────────────────────┘    └─────────────────────────────────────┘
                                              → Menu가 Viewer 자리를 흡수
```

**애니메이션 — 각 태두리별 0.1s easeOutBack**

리플로우로 인한 크기 변화는 창의 **4개 태두리(위·아래·좌·우) 각각**이 독립적으로 `0.1s easeOutBack`로 새 위치로 이동한다. 4변이 한꺼번에 통째로 스케일되는 게 아니라, **각 변이 자기 길이 변화만큼을 독립 애니메이션**한다.

```
        t=0                          t=0.1s
        ┌─ top ─┐                    ┌─── top ────┐
   left ┤       ┤ right       left ─── ┤             ┤ ─── right
        │       │          ──►         │             │
        │       │                     │             │
   left ┤       ┤ right       left ─── ┤             ┤ ─── right
        └─ bot ─┘                    └─── bot ────┘
       (작은 창)                         (커진 창)

각 변(top/bottom/left/right)이 자기 새 위치로 0.1s easeOutBack
```

핵심:
- 4변 모두 **같은 duration·easing** (`0.1s easeOutBack`)을 갖지만, **시작·종료 좌표는 각자 다르다** (어느 변이 어떻게 움직였는지에 따라).
- easeOutBack이라 살짝 "튀어나왔다 안착"하는 느낌. 단순 보간보다 자연스럽다.
- 같은 tiling space 내의 다른 동적 창 리플로우도 **동시에** 같은 규칙으로 진행한다 (동기화된 시각 효과).
- 정적·플로팅 창은 리플로우 대상이 아니므로 애니메이션 없음.
- §7.1 Enter/Exit(창의 등장과 퇴장)과 **별개** 동작이다. Reflow는 이미 떠 있는 창의 크기만 바꾼다.

## 8. 다른 디자인 언어와의 조합

BLP TILE은 **컨테이너 디자인 언어**다. 단독으로 쓰면 "회색 테두리만 있는 빈 창"만 만들 수 있다. 반드시 다른 디자인 언어가 함께 와야 완성된다.

### 8.1 조합 패턴

| 패턴 | 설명 | 예시 |
|---|---|---|
| **BLP TILE + α** | 한 가지 디자인 언어로 모든 창의 안쪽을 채움 | BLP TILE + shadcn/ui |
| **BLP TILE + α + β** | 영역별로 다른 디자인 언어를 혼용 | 상단바는 사내 가이드, 본문은 Material, 모달은 Ant |
| **BLP TILE + 통째 import** | 창 안의 콘텐츠가 외부 라이브러리 통째 | Leaflet 맵, Monaco 에디터, PDF 뷰어 |

어떤 조합이든 **창 *바깥*(타입, 위치, gap, border)만 BLP TILE이 책임**지고, **창 *안쪽*은 전적으로 다른 디자인 언어가 책임**진다. 책임 경계가 흐려지지 않게 한다.

### 8.2 좋은 인테이크 예

> "이 앱의 메인 화면은 BLP TILE + shadcn/ui로 가자.
> 동적 창 3개(상단바 / 사이드바 / 메인) + 정적 창 1개(우측 미니맵).
> 그 위에 완전 플로팅으로 로그인 모달은 Ant Design Mobile을 가져다 쓴다."

이 한 문장에 BLP TILE(레이아웃) + shadcn(메인) + Ant(모달)의 책임이 명확히 갈린다.

## 10. 구현 노트 (Reference Sketch)

CSS Grid + 약간의 상태 관리로 대부분 커버된다. 의사코드:

```html
<div class="blp-tiling-space" data-orientation="horizontal">
  <div class="blp-window" data-type="dynamic" data-weight="1" style="--focus: 0">
    <!-- 다른 디자인 언어로 채우는 영역 -->
  </div>
  <div class="blp-window" data-type="dynamic" data-weight="2" style="--focus: 1">
    …
  </div>
  <div class="blp-window" data-type="static" data-x="800" data-y="64" data-w="240" data-h="400">
    …
  </div>
</div>

<div class="blp-window" data-type="full-floating" data-anchor="center">
  <!-- 로그인 모달 -->
</div>
```

```css
.blp-tiling-space {
  display: grid;
  grid-template-columns: 1fr 2fr;
  gap: 4px;                          /* --blp-gap */
  width: 100%; height: 100%;
}
.blp-window {
  border: 4px solid var(--blp-border-idle);
  border-radius: var(--blp-radius);  /* 12px */
  outline-offset: -4px;              /* 테두리를 박스 모델 안에 포함 */
  transition: border-color 80ms linear;
}
.blp-window[style*="--focus: 1"] {
  border-color: var(--blp-border-focus);
}
.blp-window[data-type="static"]      { /* grid 밖의 절대 위치, 형제를 회피 */ }
.blp-window[data-type="fixed-float"] { position: absolute; z-index: 50; }
.blp-window[data-type="full-float"]  { position: fixed;    z-index: 100; }
```

> 핵심은 *창의 위치/타입을 데이터 속성으로 선언*하고, BLP TILE 런타임이 그걸 CSS Grid + absolute/fixed로 풀어내는 형태다. React/Vue/Svelte 어디서든 같은 컨셉으로 구현 가능.

## 11. 안티패턴 & 주의사항

- **창 안쪽까지 BLP TILE이 디자인하려고 하지 말 것.** 안쪽은 다른 디자인 언어의 영역이다. 테두리 색, gap, z-index까지만 건드린다.
- **모든 창을 동적으로 만들지 말 것.** "타일링인데 다 동적이면 뭐가 다르냐"가 된다. 툴 팔레트/모달/알림은 정적·플로팅으로 빼는 게 정상.
- **gap을 임의로 바꾸지 말 것.** 4px는 BLP TILE의 시그니처다. 의도적으로 다른 값을 쓰고 싶으면 명세에 "variant: compact (2px)" 같은 식으로 *언어 차원의 변형*으로 등록한다.
- **WM의 조작 기능(끄기/이동/단축키)을 기본값처럼 넣지 말 것.** 필요한지 먼저 묻고, 필요하면 명시적으로 켠다. 웹에서는 거의 항상 꺼둔 채로 쓴다.
- **close/minimize/fullscreen 버튼, title bar를 자동으로 넣지 말 것.** §5.1 — 기본값은 chrome-less. 예외는 명시적으로 선언해야만 켠다. 라이브러리가 기본 컴포넌트로 chrome을 깜빡 넣는 경우가 많은데, BLP TILE 컨텍스트에서는 명시적 opt-in이 아니면 전부 제거한다.
- **테두리 두께를 콘텐츠 레이아웃 계산에서 빼지 말 것.** 4px를 박스 모델 안에 포함하는 게 tiling 정확도를 지킨다. `box-sizing: border-box` + 4px border가 정답.
- **다른 디자인 언어 없이 단독 사용하지 말 것.** 그러면 회색 테두리 박스만 남는다. 항상 동반자를 정한다.

## 12. 한 줄 요약

> **BLP TILE = "어떤 디자인 언어든 담을 수 있는, 4px gap으로 맞물린 tiling 컨테이너."**
> 바깥(타입·배치·gap·테두리)은 BLP TILE이, 안(콘텐츠·컴포넌트·비주얼)은 다른 디자인 언어가.
